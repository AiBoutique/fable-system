# Refresh the fable kit (src\ + FableSetup.exe)

The kit is the repo's `src\` tree — the canonical PUBLIC copy of the fable system — plus `FableSetup.exe`, a self-contained installer with the whole kit embedded, living in the container's `EXE\` folder under the three-folder rule (repo root in a plain checkout). The repo is the fable project folder — wherever this repository is checked out. Rebuild from live files — never from memory of what they contain. The zip model is retired (r19): a `fable-restore-kit-*.zip` found anywhere is a superseded artifact to flag, never something to rebuild.

1. **Sync live → src.** Diff each live SYSTEM file against its src counterpart and copy live → src where they differ: `~\.claude\CLAUDE.md` → `src\claude-home\CLAUDE.md`; `~\.claude\settings.json` → `src\claude-home\settings.json` (discipline content only, never a blind copy — see the settings.json special-case in the direction note below); the fable-mode skill tree (SKILL.md + `references\gold-standards.md`); the refresh-kit, invest-research, and organize SKILL.md files; `~\.claude\scheduled-tasks\fable-health-check\SKILL.md` → `src\claude-home\scheduled-tasks\fable-health-check\SKILL.md`. **Direction is not automatic — decide it per file, and never by mtime** (git does not preserve mtimes: a fresh clone/checkout/`git restore` stamps src files "now"). The reliable first check is **`git status` in the repo: any uncommitted src edit means src carries changes not yet deployed to live** — capturing live→src would revert them, so deploy those src→live FIRST (below), then capture. For a file where src is clean but still differs from live, decide by CONTENT — read the actual live-vs-src hunks: a live-side edit (the user changed their `~\.claude`) ports live→src, the normal capture; a src-side change committed but never deployed (the r26-class hazard) ports src→live. An install ledger (`fable-install-ledger.json`) settles it deterministically where present AND conclusive (live==ledger ∧ src≠ledger ⇒ src is ahead; live≠ledger ∧ src≠ledger ⇒ both changed → inconclusive); where the ledger is absent or inconclusive and direction is genuinely ambiguous, show the diff and ASK — never blind-copy. Deploy src→live via the **INSTALLER** (backs up to `fable-install-backup-<stamp>`, then merges), never a plain overwrite of live `~\.claude` (no version control to recover from) — and note the installer overwrites EVERY differing live file in one run, so first port any live-only hunks in OTHER files back into src or that same run reverts them. **settings.json is never blind-copied either direction:** hand-port a genuine live hook/discipline improvement into src FIRST, then deploy through the installer's merge — the merge preserves foreign keys and personal prefs but REPLACES a fable hook with src's version, so src must already carry the improvement or it's lost (the personal keys `theme`/`tui`/`autoUpdatesChannel`/`skipWorkflowUsageWarning` stay live-only regardless). Check for newer live skills before staging and decide per-skill: system (sync) or personal (exclude). PERSONAL OVERLAYS never sync automatically — project memory NEVER enters src or the exe (public-kit policy, r19); personal UI prefs in `settings.json` (e.g. a `tui` mode) stay out of src, which ships only the three benign prefs + three hooks (settings.json is compared on discipline content minus the personal-overlay keys `theme`/`tui`/`autoUpdatesChannel`/`skipWorkflowUsageWarning` — the same carve-out the health-check's audit 4(a) and the amplification regression gate's config-drift rung use); and if the live invest-research profile line or gold-standards exemplars carry personal content, port the change by hand in genericized form.
2. **Scrub gate — before anything builds.** First purge regenerable caches from the repo (`__pycache__\` dirs, `*.pyc`): Python bytecode embeds absolute machine paths and REGROWS on every harness run — purge, don't just rely on `.gitignore` (r29 lesson: ignored bytecode still leaks via any raw-folder or zip copy of the tree). Then sweep the ENTIRE repo tree — every top-level folder (`src\`, `eval\`, `amplification\`, `expertise-system\`, `docs\`, root files; a NEW top-level folder is in scope by default; sweep text AND binary files — string-scan binaries, since a text-only grep misses embedded paths) — on two layers: **(a) identifiers** — the machine's `$env:USERNAME` and `$env:COMPUTERNAME`, the values of `git config user.name` and `user.email`, and any personal org handles you know of (derive the hunt list from the live machine, never hardcode it into this public file); **(b) classes** — any email-shaped string, any user path (`C:\Users\<x>`, `/home/<x>`, `/Users/<x>`, UNC `\\host\share`), and credential shapes (`AKIA`/`sk-`/`ghp_`/`xox`/`AIza`/`ya29`, `-----BEGIN … PRIVATE KEY`, bearer/token/password assignments). Exactly two exemption classes: upstream license attribution (`eval\LICENSE-upstream` + provenance lines — a license obligation, never scrubbed) and the installer's own synthetic scrub-test fixtures (the S7 fixture values in `src\tools\run-selftest.ps1` and the pattern lists in `src\tools\export-mcp-template.js` — fake values the selftest asserts get REDACTED; fakeness is CHECKABLE, not assumed: every fixture value must carry a self-describing marker — `LEAK`/`SECRET`/`Sup3r`-style names or an `example.com` host — an UNMARKED high-entropy value in those files is a hit, and new fixture values get diffed against the previous release before exemption). Then the **git layer** — a file sweep cannot see commit metadata. Two carriers, not one. **(i) Reflogs and git internals:** `.git\logs\HEAD` and `.git\logs\refs\**` record the ORIGINAL author identity and LOCAL-TZ offsets of every pre-scrub commit, and a history re-root does NOT rewrite them; `.git\config` may also carry a name/email. These never travel over `git push`, so they are harmless for a push-based publish — but a RAW FOLDER OR ZIP COPY of the repo carries them verbatim, so any such copy must either exclude `.git\` or run `git reflog expire --expire=now --all && git gc --prune=now` first (that is a history-metadata deletion: get approval and name the recovery path before running it). Grep `.git\logs` for the identifier hunt list as part of this step. **(ii) Commit metadata:** `git log --format="%an|%ae|%cn|%ce|%ad|%cd" --date=iso-strict` over every commit on `master` must show only the neutral publish identity in BOTH author and committer fields (committer contamination is the classic history-rewrite miss) and UTC date offsets — `iso-strict` prints UTC as `Z`, so accept `Z` or `+00:00`, nothing else (a local-TZ offset is a coarse location signal and becomes immutable on push; for new commits export `GIT_AUTHOR_DATE`/`GIT_COMMITTER_DATE` in UTC before committing). ANY other hit aborts the refresh. Paste the decisive lines in the report — identifiers hunted, classes swept, folders scanned, git layer checked, hits=0 (or the abort) — a refresh report without them didn't scrub.
3. **Regenerate the plugins snapshot**: `node src\tools\export-plugins.js "$env:USERPROFILE\.claude.json" "$env:USERPROFILE\.claude\plugins\known_marketplaces.json" src\plugins-snapshot.json` (marketplace sources + plugin names only) — PowerShell syntax, since `%USERPROFILE%` expands in cmd.exe only and node would receive the literal string. The MCP template is NOT regenerated: the public kit ships `src\mcp-servers.template.json` EMPTY — never export the machine's real server list into src.
4. **Src-resident doc stamps — BEFORE the manifest** (r24 lesson: stamped after the manifest they invalidate it and force a re-manifest + rebuild): when the system changed materially, update `src\RESTORE.md` and `src\restore-walkthrough.html` NOW (header stamp, footer, version-labeled body text — classifier vN, kit rN, assertion counts, move counts). Root-level docs live outside `src\` and wait for step 9.
5. **Manifest** (nothing in src may change after it): `powershell -NoProfile -ExecutionPolicy Bypass -File src\tools\make-manifest.ps1 -KitRoot src -KitVersion rN` (without the Bypass flag the stock execution policy refuses the script).
6. **Selftest on EVERY refresh**: `pwsh -File src\tools\run-selftest.ps1 -Kit src` (needs PowerShell 7 + Node.js + Git Bash + `claude` on PATH; installs only into disposable %TEMP%\fbt sandboxes). Require the decisive line `TOTAL: 182 passed, 0 failed` — the count is the contract; a failed selftest aborts the refresh, and an intentional assertion-count change updates this line plus the src doc stamps in step 4 (then the manifest again) and every root doc in step 9 (README, CHANGELOG, and the `docs\` pages — grep the whole repo for the OLD count to catch them all).
7. **Build the exe**: `pwsh -File src\tools\build-exe.ps1` — refuses a manifest-less src, embeds src as the payload, CRC-reads every payload entry, writes the exe replacing the old one (single-exe policy) — under the three-folder container rule build it to the sibling `EXE\` folder: `pwsh -File src\tools\build-exe.ps1 -OutFile ..\EXE\FableSetup.exe` (a plain checkout without the layout builds to the repo root, which is gitignored).
8. **Verify the shipped exe end-to-end** (S12 tested a temp build; the shipped artifact gets its own run): run the built exe (here: `..\EXE\FableSetup.exe`) with `-Unattended -TargetHome <fresh %TEMP% sandbox>` → exit 0, output contains "INSTALL PASSED", the installed CLAUDE.md hash equals `src\claude-home\CLAUDE.md`'s, and `.claude\fable-install-ledger.json` exists in the sandbox with this kit's version. Also confirm the exe's own metadata is identifier-free: `(Get-Item ..\EXE\FableSetup.exe).VersionInfo` string fields must be empty or neutral (no name, company, or machine path). Delete the sandbox afterwards — then persist the proof the deleted sandbox can't: append one line to the repo-root `build-receipts.jsonl` — `{"kit":"rN","exeSha256":"…","exeBytes":…,"selftest":"TOTAL: … passed, 0 failed","sandbox":"INSTALL PASSED","versioninfo":"clean","date":"YYYY-MM-DD"}` — the durable record that THIS exe was end-to-end verified (before r29, no evidence of the sandbox run survived its deletion).
9. **Root docs in the same pass** when the system changed materially: the root `README.md`, a `CHANGELOG.md` entry for rN, and the live current-state pages under `docs\` (`OVERVIEW.md`, `index.html` — assertion counts, feature lists, kit rN; leave dated historical files like `session-handoff-*` and the dossier alone) — all outside `src\`, so they never stale the manifest. A final repo-wide grep for `r(N-1)` and EVERY count that changed — the installer selftest count (`run-selftest.ps1` TOTAL), the amplification harness self-test count (`selftest.py`), and the `verify_answers.py` facts count — catches any straggler; these live in different files (installer count in README/RESTORE/walkthrough/docs; harness counts in `amplification\README.md`, `docs\OVERVIEW.md`, `docs\index.html`), so grep each old number repo-wide, not just the one you think you changed. **Amplification-harness edits count as "material" for THIS step even when they don't bump rN** (the harness is not in the kit manifest, so a harness-only round ships as "kit unchanged" — but if it changed a self-test or facts count, the `docs\` pages and `amplification\README.md` still carry that number and must be re-stamped in the same round, or they stale until the next rN refresh).
10. **Memory entries updated to rN before finishing** (MEMORY.md index line + the fable-system project memory's CURRENT STATE) — memory is machine-local state, never kit content — and mirror the live memory dir into the container's `private\memory\` ADDITIVELY (the machine-local personal backup; `private\` is never uploaded) — never delete a `private\memory\` file just because it's absent from live: a memory pruned live must SURVIVE in the backup, or the mirror becomes an accomplice to irreversible loss; commit the `private\` repo after mirroring so the snapshot is durable (invoking /refresh-kit IS the explicit ask for that one snapshot commit — it is the step's whole point; no other commit is implied, and the `sharable\` repo still follows the session's normal commit rules).
11. **Remind the user — every time**: the off-machine copy (a repo push or a copy of FableSetup.exe) is MANUAL by their standing choice; and since r19 project memory has NO automated backup — until they set one up, `~\.claude\projects\...\memory` lives only on this machine.
