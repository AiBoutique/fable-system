The Fable System — Restore Walkthrough

Rebuild the full fable-system setup — frontier-grade process from any executor model — on a fresh machine in about ten minutes.

01What this system is

A set of coupled artifacts that push every Claude Code session on this machine toward frontier-grade process — falsifiable definitions of done, hypothesis-driven debugging, adversarial verification, and judgment routed to the strongest available model tier. Each layer covers another layer's documented failure mode:

Two storage rules. Keep a copy of the kit off-machine (a repo push, cloud, or USB copy of FableSetup.exe — a backup on C:\ dies with C:\). And never leave a literal .claude\skills\ tree inside any folder you open Claude in — it registers duplicate scoped skills (observed live during the build). The kit's payload directory is named claude-home\ precisely so the repo itself is inert; the installer maps it to ~\.claude\.
One-click restore. Run FableSetup.exe (from the EXE\ folder beside the repo, a Release download, or your off-machine copy) — the whole kit is embedded: it extracts to a private %TEMP% staging dir, runs the installer with your arguments passed through, propagates the exit code, and cleans up. From a repo checkout, src\INSTALL.bat runs the same installer: set-equality SHA-256 integrity gate (altered, missing, and unlisted extra files all refuse the install), prerequisite checks (with winget install offers), copy-with-backup, hook merge into any existing settings.json (foreign keys and hooks preserved, validation-gated with rollback), MCP re-registration from the template (ships empty in the public kit — populate it locally), and a final PASS/FAIL self-verification table. Idempotent; never touches credentials; no personal data ships (no project memory, empty MCP template). It also cleans up after itself: its own %TEMP% logs rotate to the newest 10, stale extraction dirs go, and old install backups get a report plus an opt-in prune (default No, the 3 creation-newest always kept). Upgrades are self-cleaning from r25 on: an install ledger (.claude\fable-install-ledger.json) records what each kit version installed, and the next install removes files a previous r25+ kit left behind — only while byte-identical to what that install wrote (backed up first); anything you modified or created is kept. (Prospective by design: the first r25 install writes a baseline ledger only, so a one-time r24→r25 upgrade cleans nothing.) The numbered steps below remain as the manual fallback and as documentation of exactly what the installer does. Off-machine storage is manual by owner choice: after every /refresh-kit, copy the fresh FableSetup.exe (or push the repo) off C:\ yourself.

02Prerequisites

  • Claude Code installed and logged in (CLI or desktop app).
  • Git for Windows — hooks run in Git Bash. Without it they fall back to PowerShell: the SessionStart and SubagentStart commands survive (both are valid PowerShell — execution-verified), but the classifier is bash-syntax and shows a harmless-but-noisy per-prompt error until Git Bash exists.
  • Node.js — the classifier parses the prompt field with it. Missing → the classifier goes inactive and prints one DEGRADED repair notice per prompt until Node is restored (never the old whole-payload false-fire mode).
  • The kit: a checkout (or copy) of the repo, or just FableSetup.exe from your off-machine copy.
  • Or skip the checklist: the installer probes all three prerequisites itself and offers winget installs for anything missing.

03Restore steps

  1. One click: run FableSetup.exe

    Double-click it (or run FableSetup.exe -Unattended). It installs and verifies everything the steps below describe — continue at step 5. The exe is unsigned, so Windows SmartScreen may warn on first run ("More info" → "Run anyway"); building it yourself from src\ sidesteps that. No exe at hand? A repo checkout's src\INSTALL.bat does the same. Steps 2–4 are the fully manual fallback.

  2. Manual fallback: copy the law, the skills, and the scheduled-task prompt

    <repo> is your checkout; the kit's claude-home\ maps to ~\.claude\.

    Copy-Item "<repo>\src\claude-home\CLAUDE.md" "$HOME\.claude\CLAUDE.md"
Copy-Item "<repo>\src\claude-home\skills" "$HOME\.claude\" -Recurse -Force
Copy-Item "<repo>\src\claude-home\scheduled-tasks" "$HOME\.claude\" -Recurse -Force
  3. Manual fallback: install the hooks

    Fresh install with no $HOME\.claude\settings.json yet → copy the kit's file whole (it also carries three benign prefs: update channel, theme, workflow-warning skip):

    Copy-Item "<repo>\src\claude-home\settings.json" "$HOME\.claude\settings.json"

    If a settings.json already exists, merge only the "hooks" block from the kit into it — or hand both files to Claude and say "merge the kit's hooks into my settings.json."

  4. Project memory — yours to restore, if you kept a private backup

    Since r19 the public kit carries NO project memory (personal data never ships in the kit). If you keep your own backup, open one session in your project folder first (the projects\<slug>\memory\ directory appears), then copy your files into it.

  5. Re-create the monthly health check

    The kit installs the task's PROMPT file (.claude\scheduled-tasks\fable-health-check\SKILL.md, via step 2 or the installer), but the schedule itself lives in the app's own data and is NOT restored by file copy — verified against the scheduler on 2026-07-07. In any new session say "Create a monthly scheduled task named fable-health-check, 1st of the month at 09:00, using the prompt file at .claude\scheduled-tasks\fable-health-check\SKILL.md".

    The installed SKILL.md is the single source of truth for the audit prompt (preflight, pair consistency, hook behavior, adoption + conversion, kit freshness, memory scope integrity). Earlier walkthroughs embedded a pasteable copy here; it drifted stale against the live file and was retired in r15 — if you ever need the text, open the installed file itself.

    Scheduled tasks run while the app is open; a missed run fires on next launch. Click "Run now" once after creating it to pre-approve its tool permissions.

04Verify — five checks, two minutes, in a NEW session

  • Type /fable-mode → the skill loads its 9-section method.
  • Ask "any standing orders?" → the SessionStart order is cited back.
  • Submit "refactor the auth module" → transcript shows UserPromptSubmit hook success: Prompt classifier…
  • Say "spawn a subagent and have it report any standing orders it sees" → it reports the verify-before-reporting order.
  • /refresh-kit appears in the skill list; after you re-create the schedule (step 5), the Scheduled section shows fable-health-check.
Or delegate the whole thing: point a fresh Claude at the repo (or at FableSetup.exe) and say "restore and verify the fable system from this kit" — src\RESTORE.md carries these same instructions in Claude-executable form.

05Maintenance contract

  • Coupled pair: a threshold change in CLAUDE.md or fable-mode SKILL.md propagates to the other (2-attempt cap · 1/2 reviewers · ≥3 flake runs · Ledger 3+).
  • Review before adopt: material edits to any layer get a fresh-eyes two-lens review (correctness; integration/safety) — every version in this kit shipped that way.
  • After any change: run /refresh-kit — it re-syncs src\, scrub-gates for personal data, re-runs the 157-assertion selftest, and rebuilds FableSetup.exe; then copy the fresh exe (or push the repo) off-machine. Project memory is NOT in the kit since r19 — keep your own backup.
  • Read the monthly health report. It is the system's only feedback loop; "ATTENTION" verdicts are the trigger for the next upgrade, not noise.
