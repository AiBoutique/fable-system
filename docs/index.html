01 — What it is

Discipline that ports; enforcement that doesn’t have to

The Fable System packages the process behind frontier-grade agent output: a global rulebook, an operating-method skill, enforcement hooks, a domain-expertise layer, an eval harness, and a self-verifying installer. It is built so the reasoning discipline is model-agnostic — a portable core — and only the enforcement layer is platform-specific — a swappable adapter. The same discipline can front Claude today and other models later.

Two commitments run through everything:

  • Measure before claiming. No rule change ships without an eval delta; nulls and failures are recorded, not hidden.
  • Scale ceremony to stakes. Heavy for high-blast-radius work; out of the way for trivial edits.

03 — Measured performance

The honest numbers

All figures are from the system’s own eval harness on Opus 4.8 as host, with claude-fable-5 as the reference bar (R). Conditions: A0 = bare model, A1 = model + core, R = reference model bare. Run logs are held privately; the aggregates below are re-derived from them and were independently accuracy-reviewed.

The headline, stated plainly

There is no measured across-the-board percentage lift — because on the tasks tested, the bare host model is already at ceiling. On an 18-task single-shot, tools-off text suite across six dimensions:

ConditionMean scoreReading
A0 — bare Opus 4.80.965the floor
A1 — + discipline core0.972the product
R — bare claude-fable-50.944the bar to match
  • Measured lift (A1 − A0) = +0.007, read as 0.000. An independent review traced the entire nominal +0.007 to one contaminated cell (a task whose answer markers appear verbatim in the injected core); excluding it, the lift is zero. The bare model had no headroom to amplify here.
  • The harness held the frontier line: A1 matched or exceeded the reference on 5 of 6 dimensions (gap = R − A1 = −0.028, N=1). Reaching the bar, not beating it by points, is the honest claim for this suite.

Where a core change did measurably help

The one place the discipline produced a measured, reproducible gain is calibration — how well the model’s stated confidence intervals contain the truth. On a held-out tight-anchor bed (4 tasks, N=3/arm):

ArmInterval hit-rateStability
base core0.8330.833
+ [0,20] confidence rule1.0001.000

+16.7 pp on interval hit-rate (0.833 → 1.000), N=3, zero regressions — the program’s first measured positive core delta. It is narrow and shape-specific (tight-anchor counting tasks), not a broad lift. Two honesty caveats:

  • A second candidate rule (mechanical interval-widening) tied at 1.000 — so the [0,20] rule is an effective fix, not the uniquely effective one; it was the one adopted, widen as runner-up.
  • The change did not clear its own pre-stated “adopt iff a majority of tasks improve” bar — 3 of 4 tasks were already at ceiling, making that rule unsatisfiable as written. Adoption was an owner decision informed by the A/B, not an automatic protocol pass: A/B-supported and owner-adopted, not A/B-mandated.

Regression safety of that change

Re-run across the full 18-task gradient (base core vs adopted core, both fresh, N=1/arm; the two cores differ by exactly the 13 added lines):

0 regressions, 0 infra-exclusions. 16 of 18 cells identical across arms; 2 cells rose but at N=1 both are run-to-run noise not attributable to a calibration rule (one is the known contaminated cell, one a base-arm miscount on an injection task the rule cannot mechanistically affect). The raw mean shift (0.91 → 0.97) is those two noisy cells — reported here as “no regressions,” never as a percentage gain.

Efficiency

The portable core reaches the full core’s benchmark behavior at ~28% of the size (≈12.8 KB vs ≈46 KB today; the measurement was taken when the full core was ≈43 KB). It measured 0.965 on the 18-task suite — equal to the full core once the single contaminated cell is excluded (the full core’s raw 0.972 is that cell). You get the discipline’s measured effect at roughly a quarter of the token cost, in a model-agnostic form.

04 — Boundaries of the claim

What we did not claim, and what isn’t measured yet

Deliberately not claimed

  • No “N% smarter across the board.” The measurement says lift ≈ 0 on single-shot tools-off text tasks because the base model is at ceiling. Inventing a percentage would contradict the system’s own evidence.
  • No blanket dimension wins. The support for “reaches the frontier bar” is A1 ≥ R on 5/6 dimensions at N=1 — stated with that N, not inflated.
  • Recorded nulls stand. Two earlier core-change candidates were rejected by their pre-stated A/B protocols; those nulls are kept on the record, not buried.

Not yet measured — where lift is plausible but unproven

The tested suite is single-shot, tools-off, and text-only — the conditions least favorable to scaffolding. The mechanisms this suite cannot exercise:

  • Tool-dependent tasks — the “do arithmetic through code” rule is inert when no tools run.
  • Multi-turn, long-horizon jobs — the ledger, compaction checkpoints, and milestone re-verification only pay off over many steps.
  • Adversarial / multi-agent verification — independent-reviewer and refutation patterns need a fleet to show their effect.
  • Cross-model porting — the portable core is measured on Claude; Grok / OpenAI runs await API keys.
Honest framing: proven to hold the frontier line and to fix calibration on single-shot text; expected-but-unproven to lift harder, tool-using, long-horizon work.

05 — How the numbers are produced

Reproducibility

  • Every code-checkable task truth is re-derived by an independent solver (verify_answers.py, 33 facts, 0 failures).
  • Graders are deterministic and report V (validity / pass-rate), S (stability across repeated runs), and A (agreement with the reference) as separate axes — agreement is never optimized alone.
  • Infra failures are excluded loudly as score: None, never scored as zeros.
  • A 280-assertion self-test and a 6-rung twice-monthly regression gate guard the machinery itself.
  • N is small (often N=1/cell) and always declared; single runs are not presented as variance-characterized results.

Any external research percentages that appear in the project’s advancement dossier are directional context about the underlying techniques, drawn from third-party literature — not measured claims about this system.

06 — Getting it running

Install

Windows, one click: run FableSetup.exe (from a GitHub Release, or build it yourself below) — the whole kit is embedded; it extracts, installs, verifies, and cleans up. Unattended: FableSetup.exe -Unattended.

From a checkout: src\INSTALL.bat — same installer, same integrity gate.

macOS / Linux: manual steps in src/restore-walkthrough.html (the hooks are bash-native; the installer is Windows-specific).

No personal data ships in the public kit — no project memory, and an empty MCP template you populate locally (the export tool redacts secrets automatically). After installing, open a new session and run the two-minute check in src/RESTORE.md.

07 — The rules earn their keep

No rule change ships without evidence

A trap-suite regression harness (8 adversarial fixtures, adopted unchanged under MIT from Sahir619/fable-method) must pass at its prior level — or a new failing trap lands first. The installer ships with its own separate 157-assertion harness (its totals live in the changelog and build receipts; the trap rounds live in the results file), run on every rebuild; results are logged in the changelog and results file, nulls and failures included.

08 — Build the exe yourself

From source, no toolchain to install

powershell -NoProfile -ExecutionPolicy Bypass -File src\tools\make-manifest.ps1 -KitRoot src -KitVersion rN
pwsh -File src\tools\run-selftest.ps1 -Kit src     # TOTAL: 157 passed, 0 failed
pwsh -File src\tools\build-exe.ps1                 # -> FableSetup.exe

The exe is a small C# stub compiled with the .NET Framework csc.exe that ships with Windows, kit embedded as a resource. It is unsigned — SmartScreen may warn on first run; building it yourself sidesteps that. The binary is untracked by git and lives in a sibling EXE\ folder under the three-folder container rule (sharable working repo / private\ / EXE\).

09 — Publishing

When you finalize

  1. History: already re-rooted. master carries a single scrubbed root commit under a neutral identity with UTC-only timestamps (r28, re-dated r29). The pre-scrub development history survives only in the local archive/pre-publish-history branch — never push it. A local pre-push hook refuses every destination except refs/heads/master, and refuses any push whose source is not the local master tip; git hooks never travel with clones, so re-create the guard in any secondary clone.
  2. License: MIT, already in the tree (LICENSE, under a neutral "Fable System authors" holder — put your public name on the copyright line if you want attribution). The bundled eval fixtures keep their upstream MIT terms.
  3. Attach FableSetup.exe to a GitHub Release — the binary is untracked by design.
  4. Push under the identity you intend to be public for the new history.
