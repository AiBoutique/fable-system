## P0 — Foundational (do first; each unblocks downstream work)

### P0.1 — Decide and execute the expertise-system's kit membership
The 25 expertise skills are installed live and versioned in `sharable/expertise-system/` but are **not** in `src/claude-home/skills/`, so the kit manifest, the 126-assertion selftest, and `FableSetup.exe` ignore them — a fresh install ships the discipline spine but none of the domain mastery. *(verified: manifest lists 4 skills; selftest expertise-grep = 0)* [axis5#1]
- **Options:** (a) **promote into the kit** — copy to `src/claude-home/skills/`, extend the selftest with per-skill present/lint assertions, add the coverage+lint audits as a selftest rung, `/refresh-kit`; (b) **separate installable** reusing `build/lint-structure.ps1`; (c) keep manual (no gate, undiscoverable). Recommend **(a)** if the expertise layer is "part of the system," **(b)** if it's a distinct product. Falsifier for (a): if selftest runtime or exe size becomes unacceptable, fall back to (b).
- **Effort:** S–L (option-dependent). **Risk:** a wrong split ships a half-system; promoting enlarges payload + selftest count and makes every expertise edit require a kit refresh.
- **Next step:** make the (a)/(b)/(c) call, then — if (a) — execute via the plugin path in P0.2, which is the cleanest integration mechanism.
- **RECOMMENDATION (2026-07-16, for the next-session Fable review — not yet executed):** resolve this via the **plugin path (P0.2), not by growing the bespoke `.exe` (standalone option a)**. Rationale, strengthened by this session's amplification work: the north-star's portable-core/adapter split (`amplification/core/portable-core.md`) makes the discipline spine AND the 24 expertise skills all model-agnostic prompt content that maps 1:1 onto a Claude Code plugin's `skills/` — the plugin is the platform-native "one install ships spine + domain mastery" answer and the open-sourcing vehicle, and it retires the exe/manifest/126-selftest rather than enlarging them. **Interim answer: (c)/(b)** — keep the expertise skills a separate/manual install; do NOT invest in promoting them into an exe that P0.2 will retire. **Do NOT unilaterally rebuild the kit for this** — it is a costly-to-reverse owner fork (payload +875KB/51 files — measured 2026-07-16; this line's earlier "+150KB" estimate was ~6× understated — selftest +dozens of assertions, exe rebuild, README/RESTORE/walkthrough/CHANGELOG churn). **Falsifier:** if the plugin path proves infeasible — specifically if skill namespacing (`/fable:organize`) breaks the classifier/hook matching on bare skill names (the P0.2 risk) and can't be worked around — fall back to (a) promote-into-exe. Decision reserved to the owner; the Fable review adjudicates with full context.

### P0.2 — Repackage the fable system as a Claude Code plugin (retire the bespoke .exe)
A plugin is a directory with `.claude-plugin/plugin.json` bundling root-level `skills/ agents/ hooks/hooks.json commands/ .mcp.json monitors/ bin/ settings.json`; validate with `claude plugin validate`; distribute via the official/community marketplaces; there's an explicit "convert existing `.claude/` config → plugin" migration path. *(verified first-hand at code.claude.com/docs/en/plugins, 2026-07-15)* [axis1#3]
- **Why:** replaces `FableSetup.exe` + hand-rolled manifest + selftest with the platform-native package the system already targets, is the natural open-sourcing vehicle (license still TBD), and simultaneously solves P0.1's integration. CLAUDE.md, fable-mode, the 24 expertise skills, hooks, and workflow skills map 1:1 onto plugin dirs.
- **Refinements (verified):** a lighter middle path exists — `claude plugin init` creates a *skills-directory plugin* (`name@skills-dir`) that auto-loads with no marketplace; plugin skills are namespaced `/fable:organize`, but the bare `/organize` keeps working if the original is retained (softens the classifier-breakage risk at the cost of duplicates); `settings.json`'s `agent` key can set the default main-thread agent.
- **Effort:** L. **Risk:** namespacing breaks any hook/classifier matching bare skill names; the git-history PII scrub still applies before any marketplace push.
- **Next step:** in a worktree, scaffold `plugin.json`, copy the live skills/agents/hooks into plugin layout, `claude plugin validate`, load via `--plugin-dir`, and diff behavior against the current install before deciding whether the plugin supersedes `refresh-kit`.
- **STATUS 2026-07-17 (expertise slice executed; spine slice still deferred):** the expertise layer is now a validated plugin — `expertise-system/.claude-plugin/plugin.json` (`fable-expertise` v0.1.0) makes `expertise-system/` a loadable plugin dir; `claude plugin validate` → passed (EXIT=0, only the intentional missing-author warning — author omitted to keep the tree PII-clean); expertise lint ALL PASS; the 25 skills' descriptions are strict-YAML-valid after the `697e1b0` quoting. **Posture decided: COEXIST** — install the plugin alongside the bare-named live skills (the adjudicator's mechanical analysis: retaining bare originals means namespacing cannot break auto-triggering, whose keys are descriptions, not names; the only bare-name matcher in the system is the spine health-check regex, which the expertise plugin never touches). **The live namespaced auto-trigger A/B PASSED (2026-07-17, Opus 4.8).** Three isolated-home cells: a bare-named control (finance prompt → the model auto-invoked the `finance-value` skill), the namespaced arm (same finance prompt + the plugin via `--plugin-dir` → auto-invoked `fable-expertise:finance-value`), and a router test (cross-domain prompt → auto-invoked `fable-expertise:expertise-atlas`), with all 25 skills present namespaced. Namespacing does **not** break auto-triggering — identical behavior bare vs namespaced, only the name is prefixed — so the mechanical falsifier is now confirmed negative at the live level and the COEXIST posture stands empirically, not just by argument. Full P0.2 (retiring the exe) stays deferred: plugins cannot carry a root project `CLAUDE.md`, settings hooks, or scheduled tasks (verified at the plugins docs, 2026-07-16), so the spine keeps its exe/manual bootstrap.

### P0.3 — Stand up an eval + continuous-verification harness (behavioral, not just structural)
The expertise system is verified *once* (coverage audit = term-presence; structure lint); there is no measure of whether a skill **improves output, triggers correctly, or has gone stale.** Anthropic's own guidance: "evaluations are your source of truth," and there is no built-in runner. The **`skill-creator` plugin is already installed on this machine** and ships the machinery: A/B with-skill-vs-baseline benchmarking with pass-rate deltas, description-trigger optimization on a held-out set, regression tracking (`history.json`), and a discrimination gate that flags non-differentiating/flaky assertions. *(verified on-disk + platform docs)* [axis3 all; axis2#6; axis5#3 — three-way convergence]
- **Method (Anthropic, verified):** 20–50 tasks drawn from real failures; prefer cheap code-based graders, LLM-judge only where calibrated; **grade the outcome not the path**; balanced should-fire/should-not-fire sets; cheat-resistant graders; read transcripts.
- **Effort:** M per skill / L for all 24 (start with 2 high-traffic skills). **Risk:** weak eval tasks give false confidence (mitigate with the discrimination gate); model-graded scores are non-deterministic (pin the grader, act on sustained drops, keep a safety-"invariant" assertion class exempt from the discrimination cull); the `run_loop` trigger-optimizer shells to `claude -p` — unproven on Windows (port to PowerShell beside `audit-coverage.ps1` if it fails).
- **Next step:** give `finance-value` and `invest-research` an `evals/evals.json` (3–5 tasks each), run the skill-creator loop once, record the with-vs-without delta as the first datapoint in a new `expertise-system/evals/` tree; then wrap `audit-coverage.ps1` + `lint-structure.ps1` + the eval subset into a weekly **scheduled-tasks** job that fails loudly on regression (the behavioral analogue of the 126-assertion installer selftest) and add a `-CheckFreshness` switch that HTTP-checks the URLs in each skill's Evidence-set/References.

### P1.1 — Bind fable's rigor tiers to the `effort` dial + model-routing settings

Claude Code `fallbackModel` (up to 3 in order) + `availableModels` keep long runs on the frontier tier. [axis2#1 + axis1#7]
- **Why:** turns the Trivial/Default/High-risk rigor tiers from prose into a settable parameter and answers "when does more effort pay" with primary guidance (raise on observed shallow reasoning; step down only when evals hold). - **Next step:** add an "effort" column to the fable-mode rigor table at Scope, require the chosen level be named in the plan/ledger, and set a frontier `fallbackModel` chain in settings.

### P1.4 — Make the fable ledger a persisted memory file with compaction checkpoints
Anthropic's context-editing + memory tool measured **+39%** over baseline on agentic search (context editing alone +29%); a 100-turn task cut tokens **84%**. - **Why:** upgrades long-horizon hygiene — the persisted file *is* the ledger; "checkpoint-to-file before compaction, resume-from-file after" earns promotion from advice to a hard rule; "all features start failing" is a concrete falsifiable-done device that reinforces the anti-manufactured-pass rules.
- **Effort:** M. **Risk:** the gains are Anthropic-internal search-task evals; portability to coding/refactor is unmeasured.
- **Next step:** add a mandatory "compaction checkpoint" section to the fable ledger template and a resume move that reads it before any post-compaction edit; add a "feature-ledger init" option (all-failing checklist + progress file committed to git) for large builds.

## P2 — Valuable once the above land

### P2.1 — Adversarial-verification refinement: reframe own output as external before self-review
The "Self-Correction Illusion" preprint (arXiv 2606.05976, 2026-06-04) holds a claim byte-identical and varies only the role label: relabeling from the model's own `<thought>` to an external role lifts explicit-correction rate **23–93 percentage points**; "the failure to self-correct is a chat-template artifact, not a cognitive deficit." *(agent-verified)* [axis2#2] — This is the evidentiary backbone for fable's adversarial multi-agent verification. **Next step:** in the "no subagent channel → re-read the diff yourself" rule, instruct re-presenting the diff as an external artifact ("a diff from another engineer") before critiquing. Effort S. Risk: single non-peer-reviewed preprint, varies by model family — keep the independent-agent path primary.

### P2.2 — Sharpen the orchestration fan-out gate
Anthropic's multi-agent research system: multi-agent beat single-agent Opus **90.2%** on research, but **token use alone explains 80% of variance**, multi-agent burns **~15× chat tokens**, and it **underperforms on shared-context/dependency work — "coding is explicitly a poor fit."** Subagents are synchronous (can't be steered mid-flight). *(agent-verified)* [axis2#5] — **Next step:** add an explicit "parallelize?" gate to fable-mode orchestration (fan out only if subtasks are independent AND breadth-first; else single-agent), record the ~15× multiplier, and brief subagents completely up front since they can't be corrected once launched. Effort S. Risk: 90.2% is internal research eval — adopt the direction, don't quote the magnitude for code.

### P2.3 — Documentation & discoverability coherence
The expertise system is invisible in every kit-level doc: `README.md` "what's inside" and `RESTORE.md` describe a 4-skill system; CHANGELOG's last entry is r22 with no 2026-07-14 entry; a published repo would ship an unexplained 80-file folder. *(verified: grep expertise in README/CHANGELOG = 0)* [axis5#2] — **Next step:** add an `expertise-system/` row to README, a CHANGELOG entry (24+1 skills, 63 topics, 4,996 terms, audits passed), and one RESTORE/walkthrough line stating whether the exe installs it (ties to P0.1). Effort M, docs-only.

### P2.4 — Deterministic Rank-0 guards at the harness layer
`sandbox.credentials` blocks sandboxed commands from reading secret files/env (v2.1.181); auto-mode blocks destructive git + IaC destroy (v2.1.183) and prompts on catastrophic `$(…)` even under skip-permissions (v2.1.207); `PreToolUse` can deny or rewrite args. *(agent-verified)* [axis1#5] — **Next step:** enable `sandbox.credentials` and add a `PreToolUse` deny rule for secret-file reads + a Bash rule mirroring the Irreversibility list; verify `refresh-kit` still runs. Claude Code memory now **errors instead of silently truncating** when the `MEMORY.md` index exceeds the read limit (v2.1.210) and exposes `CLAUDE_MEMORY_STORES`. *(agent-verified; caching claims INFERRED — verify prompt-caching TTL/pricing before adopting)* [axis2#7 + axis1#6] — **Next step:** add "prefer tool-search / programmatic tool calling for large tool sets and multi-tool fan-out" to the capabilities-inventory move; confirm the current `MEMORY.md` index sits under the read limit and add that expectation to the health-check.
